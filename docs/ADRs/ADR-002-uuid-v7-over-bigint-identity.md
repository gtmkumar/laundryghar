# ADR-002 — UUID primary keys over BIGINT identity

**Status:** Accepted (retro-documented 2026-06-10) · **Deciders:** Architecture

## Context

A multi-service system (9 services + Worker over one database) needs IDs that can be generated by any service — or by a mobile client for idempotent retries — without coordinating on a central sequence. Sequential BIGINTs also leak business volume ("order 10482") across tenants and make future sharding/merging painful.

## Decision

**UUID primary keys on every table.** All `id` columns are `UUID PRIMARY KEY DEFAULT gen_random_uuid()` (pgcrypto), so a row can always be created DB-side; the .NET services generate IDs application-side as `Guid` values, which keeps inserts coordination-free. The original spec intent was time-sortable UUID v7 (see the header of `database_scripts/00_kernel.sql`); the deployed default `gen_random_uuid()` is v4, so tables that need time-ordered scans rely on explicit `created_at`/`occurred_at` indexes instead of PK ordering. Human-facing sequence numbers (order numbers, invoice numbers) are a separate concern, served by dedicated per-tenant sequence tables (`order_lifecycle.order_number_sequences`, `order_lifecycle.invoice_number_sequences` — `db/patches/order_number_sequence.sql`, `db/patches/invoice_generation.sql`).

**Where it lives:** every `CREATE TABLE` in `database_scripts/0*_*.sql`; FK columns are all UUID.

## Consequences

- **+** Any service or client can mint an ID; idempotent create flows are natural.
- **+** No enumerable IDs across tenants; no sequence contention.
- **−** 16-byte keys → larger indexes than BIGINT; random v4 inserts scatter the PK index (accepted; hot tables are time-partitioned anyway, per ADR-004).
- **−** Users never see raw UUIDs — display numbers come from the sequence tables.
