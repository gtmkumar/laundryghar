**Non-obvious invariants (FULL-11):**
- `kernel.outbox_events` RLS is INERT (rls_proposal.sql never runs ENABLE ROW LEVEL SECURITY on it), so a partner-session producer with no brand context can INSERT the outbox row â€” that's why the cross-BC emit works from the operations partner lane.
- Two consumers drain the same outbox rows WITHOUT racing because they touch DISJOINT state: OutboxEventRelayService exclusively owns Status/PublishedAt (generic broker relay), while cursor-based domain consumers (LoyaltyEarnService, PartnerBookingDebitService) track a per-consumer NotificationEventCursor watermark and NEVER mutate Status. Any NEW outbox domain-consumer MUST follow the cursor pattern and leave Status alone, or it will race the relay. This is the mandated template.
- The debit consumer dispatches via IDispatcher resolved from the worker scope (CreateWorkerAsyncScope â†’ BypassRls); DebitPartnerWalletCommand takes PartnerId EXPLICITLY (not from tenant) precisely so the RLS-bypassed worker targets the booking's partner.

**How to apply:**
- Load-bearing invariant: the `partners` ROOT table's `rls_partner` keys on `id = current_partner_id()` (a partner session reads only its OWN org row). Provisioning a NEW partner + its first `partner_admin` has no current_partner_id yet, so it MUST run under `app.bypass_rls = true` (platform-admin path) â€” same treatment as the admin-only `tenancy_org.brands`/`platforms`. Don't expect a plain app_user session to insert a partner.
- `partner_users`/`partner_bookings` isolate by their `partner_id` column (standard rls_brand-shape).
- RLS enforces only when the app connects as `app_user` (non-owner); postgres/superuser bypasses natively. rls_partner.sql ENABLES RLS immediately (unlike inert rls_proposal.sql).
- partner_bookings.brand_id is a soft reference (no FK) â€” brands are admin-only/cross-schema.
- MVP-6 auth+booking endpoints (done): partner OTP login lives on the CORE host (the token issuer) at /api/v1/partner/auth (send+verify, anon, purpose=`partner_login`); bookings live on OPERATIONS at /api/v1/partner/bookings (PartnerOnly). Verify reuses the StepUp OTP crypto (SEC1 lockout + SEC2 HMAC), then mints via CreatePartnerAccessToken â€” no refresh token in MVP. Booking create sources partner_id from ICurrentTenant.PartnerId (NOT the request) so rls_partner WITH CHECK can't be spoofed; created_by = JWT sub via ICurrentUser.UserId.
- **Non-obvious gotcha:** the anonymous partner OTP send/verify paths query `partner_users`/`partners`, which have rls_partner ENABLED â€” but there's no partner context yet (pre-auth). This mirrors how customer/staff OTP+refresh paths are allow-listed. Any new pre-auth partner lookup path needs the same treatment.
